    OpenMP实现并行的三个关键步骤

    1. 划分并行区域
    2. 分配线程数量
    3. 分配任务

    划分并行区域

    目的：指定一个并行执行的代码块，并为之划分相应数量的线程

    指令parallel

    建立并行区域，确定线程数量并创建线程组

    设定线程数量

    默认模式

    不做显式声明，扩展性好

    静态模式

    omp_set_num_threads(n), 直接指定n个线程

    动态模式

    omp_set_dynamic()，依据实际需求分配

    嵌套模式：omp_set_nest()

    并行区域内，继续划分并行区域
    (不理解这么做的实际应用价值)

    条件模式：if()

    当if语句可满足时，执行并行区域

    线程数量过多而任务少时(计算负载小)，切换线程的耗时高于执行任务耗时

    任务分配

    在划分出并行区域后，我们应考虑如何将任务分配给不同的并行区域，能最大化利用计算机性能

    Why? 为什么需要任务分配

    原因一：不同任务计算量不同

    考虑以下循环

    #pragma omp parallel for
for (int i=0; i<100000; i++)
    temp=i*i;

    如果简单地将任务按数量均匀分配给多个线程，考虑负责i=99999的线程，需计算99999*99999，而分配到i=2的线程仅需计算2*2，此两者线程负载是显著不同的。

    因此，直接按任务数量分配是不可行的，因为无法保证每个任务的计算量一致

    原因二：划分的误差随核数增多而增大

    假设有一个执行时间为16个时间单位的程序，如果划分误差为1个时间单位:
    4核CPU上加速比16/(4+1)=3.2，是理想加速比的3.2/4=80%
    16核CPU上加速比16/(1+1)=8，是理想加速比的8/16=50%
    随着核心数增加，效率下降的比例更加明显

    How? 如何实现任务分配

    循环：schedule子句

    schedule(static,size)
    schedule(dynamic,size)
    schedule(guide,size)

    static：静态调度，朴素地按任务数量划分
    dynamic：动态调度，运行按实际情况使分配任务
    guide：类似动态调度，但刚开始划分的任务块较大，随后逐渐减小

    动态调度会产生额外开销，但可以依据实际情况平衡负载

    非循环：sections

    #pragma omp parallel sections
{
    #pragma omp section {...}
    #pragma omp section {...}
}

    section嵌套在sections划分的并行区中，每个section仅被线程组中的线程执行一次(如果把parallel for循环显式展开，可以等效出一个sections形式)

    单个线程：single

    #pragma omp single nowait

    single指明代码块仅由单个线程执行，存在隐形同步栅栏(其余线程需等待single块执行完毕)
    如果声明nowait，则其余线程可绕过single代码块继续执行

    负载平衡

    对于多核CPU而言，我们希望在同一时刻，尽可能多的核心在一起工作
    不希望存在空闲的核心(老资本家了😁)

    划分策略

    静态负载平衡

    人为划分工作任务，但无法得任务的实际运行耗时，无法保证实际运行时各个线程耗时相同

    动态负载平衡

    在任务执行过程中动态分配，人工设定调度策略后由编译器实现

    依赖关系

    并行需在同一时刻执行所有线程，需保证以任意顺序执行线程时，最终的结果一致
    但如果线程之间存在依赖关系，不同顺序执行线程会导致不同的结果
    因此，尽可能地消除依赖是并行的前提

    循环依赖

    读后写

    for (int i=0; i<n; i++)
    a[i]=a[i+1];

//修改方式，创建一个数组a[]的副本
//副本在迭代中不会被修改，永远是原始值
for (int i=0; i<n; i++)
    a[i]=a_copy[i+1]

    a[i]的值取决于a[i+1]，因此计算a[i]时需保证a[i+1]的值未被修改
    对于a[i+1]而言，需“先读后写”，但并行时难以保证
    可通过创建副本消除依赖

    写后写

    for (int i=0; i<n; i++)
{
    a[i]=b[i];
    a[i+1]=c[i];
    //修改建议: 提供副本存储每次迭代的a[i]
    a_copy[i]=b[i];
}

    此处前两次迭代分别处理a[0],a[1]及a[1],a[2]
    这两次迭代均对a[1]进行赋值，因此a[1]最终值取决于两次迭代的顺序，不能直接并行
    同样的，可通过创建副本消除依赖

    写后读

    for (int i=1; i<n; i++)
{
    b[i]=b[i]+a[i-1];
    a[i]=a[i]+c[i];
}

//重构代码以消除依赖
b[1]=b[1]+a[0];
for (int i=1; i<n-1; i++)
{
    a[i]=a[i]+c[i];
    b[i+1]=b[i+1]+a[i];
}
a[n]=a[n]+c[n]

    上述修改前代码中，b[i]的值取决于a[i-1]的值，即第i次迭代依赖于第i-1次迭代(必须保证i-1次迭代先完成)
    可通过重构代码消除依赖
