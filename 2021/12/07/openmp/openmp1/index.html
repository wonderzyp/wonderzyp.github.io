OpenMP并行与性能提升

2021-12-07
1.9k- 7m-
HPC-高性能计算OpenMP

本文主要内容：

  1. 使用OpenMP的通用步骤
  2. 任务task的概念
  3. 依赖关系(读后读、写后写、读后写)
  4. 执行不必要的工作实现分支并行(这个思想很有意思)

1 OpenMP并行的常用步骤

  1. 划分并行区域
  2. 分配任务

1.1 划分并行区域

指定一个并行运行的代码块，并为其分配相应数量的线程

1.1.1 并行区域

通过parallel指令划分
常见有for, sections和tasks三种并行类型

#pragma omp parallel
{
    //此区域内代码并行执行
    #pragma omp for {...}
    #pragma omp sections{section{...}, section{...}...}
}

for：循环结构

//下面语句等效为 划分并行区域+指定循环类型
#pragma omp parallel for

sections：非循环结构

#pragma omp parallel sections
{
    //多个section嵌套在sections内
    #pragma omp section{...} 
}

single：单个线程

#pragma omp single nowait

single指明代码块仅由单个线程执行，存在隐形同步栅栏(其余线程需等待single块执行完毕)
若声明nowait，则其余线程可绕过single代码块继续执行

tasks：任务

不同于指令for和sections，task可动态定义任务
线程遇到task指令后开始创建任务，当创建的未分配线程的任务数量达到临界值时，停止创建任务并开始执行任务
一旦未分配的任务数量足够低时，线程再次开始创建任务

引入task的原因：
相较于for与section，task可并行实现不规则循环(如do while)与递归函数

//由于b[i]的值取决于实际运行，parallel for无法并行下列代码
#pragma omp parallel for
for(int i=1; i<m; i+=b[i])
    {...}

//task实现递归
int fib(int n)
{
    int result,i,j;
    if (n<2) result=n;
    else
    {
        #pragma omp task shared(i) firstprivate(n) final(n<=10) mergeable
        i=fib(n-1);
        #pragma omp task shared(j) firstprivate(n) final(n<=10) mergeable
        j=fib(n-2);
        #pragma omp taskwait //确保计算f(n)时，f(n-1)与f(n-2)已完成计算
        result=i+j;
    }
    return result;
}

int main()
{
    #pragma omp parallel shared(n) 
    {
        ans=fib(n);
        #pragma omp single
        printf("the ans is %d\n", ans);
    }
}

1.1.2 分配线程

分配线程有两类方式：

  1. 静态分配：人工显式指定，线程数量固定
  2. 动态分配：系统按实际任务自行动态分配

注：线程数量过多但计算负载小时，切换线程的耗时高于执行任务耗时，导致并行效率低下

1.2 工作分配与负载平衡

在划分出并行区域后，需考虑如何将任务分配给不同的并行区域
基本原则：同一时刻，尽可能多的CPU核心同时工作，避免存在空闲核心(老资本家了😁)

1.2.1 为什么需要负载平衡

原因一：不同迭代次数的计算量不同

考虑以下循环

#pragma omp parallel for
for (int i=0; i<100000; i++)
    temp=i*i;

如果简单地将任务按数量均匀分配给多个线程，考虑负责i=99999的线程，需计算99999*99999，而分配到i=2的线程仅需计算2*2，此两者线程负载是显著不同的。

因此，直接按任务数量分配是不可行的，因为无法保证每个任务的计算量一致

原因二：划分的误差随核数增多而增大

假设有一个执行时间为16个时间单位的程序，如果划分误差为1个时间单位:
4核CPU上加速比16/(4+1)=3.2，是理想加速比的3.2/4=80%
16核CPU上加速比16/(1+1)=8，是理想加速比的8/16=50%

随着核心数增加，效率下降的比例更加明显

1.2.2 循环工作量的划分

  1. 静态负载平衡：人工划分工作，但无法得到任务实际运行耗时，因此无法保证实际运行时各个线程耗时相同
  2. 动态负载平衡：在任务执行过程中动态分配，人工设定调度策略后由编译器实现

策略分类：
static：静态调度，朴素地按任务数量划分
dynamic：动态调度，运行按实际情况使分配任务
guide：类似动态调度，但刚开始划分的任务块较大，随后逐渐减小

//schedule("类型","大小")
schedule(static,size)

动态调度会产生额外开销，但可以依据实际情况平衡负载

2 依赖关系

并行在同一时刻执行所有线程，需保证以任意顺序执行线程时，最终的结果一致
但如果线程之间存在依赖关系，不同顺序执行线程会导致不同的结果
因此，尽可能地消除依赖是并行的前提

2.1 循环依赖

2.1.1 读后写

读取元素A之后，下一次迭代会改变A的值
需保证先读后写，读到未被修改的初始值

for (int i=0; i<n; i++)
    a[i]=a[i+1];

//修改方式，创建一个数组a[]的副本
//副本在迭代中不会被修改，永远是原始值
for (int i=0; i<n; i++)
    a[i]=a_copy[i+1]

a[i]的值取决于a[i+1]，因此计算a[i]时需保证a[i+1]的值未被修改
对于a[i+1]而言，需“先读后写”，但并行时难以保证
可通过创建副本消除依赖

2.1.2 写后写

对元素A进行多次写入操作，元素A的最终值取决于最后一次写入操作

for (int i=0; i<n; i++)
{
    a[i]=b[i];
    a[i+1]=c[i];
    //修改建议: 提供副本存储每次迭代的a[i]
    a_copy[i]=b[i];
}

此处前两次迭代分别处理a[0],a[1]及a[1],a[2]
这两次迭代均对a[1]进行赋值，因此a[1]最终值取决于两次迭代的顺序，不能直接并行
同样的，可通过创建副本消除依赖

2.1.3 写后读

先更改元素A的值再读取，必须保证写操作先完成，否则读到A的旧值

for (int i=1; i<n; i++)
{
    b[i]=b[i]+a[i-1];
    a[i]=a[i]+c[i];
}
//重构代码以消除依赖
b[1]=b[1]+a[0];
for (int i=1; i<n-1; i++)
{
    a[i]=a[i]+c[i];
    b[i+1]=b[i+1]+a[i];
}
a[n]=a[n]+c[n]

上述修改前代码中，b[i]的值取决于a[i-1]的值，即第i次迭代依赖于第i-1次迭代(必须保证i-1次迭代先完成)
可通过重构代码消除依赖

2.2 任务依赖(并行if语句)

前后两个任务存在依赖关系，可通过执行不必要的工作实现并行化

int c=fun1(...);
if (c>c0) return c+fun2();
else return c;

fun1必须执行，fun2可能执行(取决于c是否大于c0)
设执行fun1需t1秒，执行fun2需t2秒，且实际执行fun2的几率为P
则串行总时间为t1+t2*P

改进思路：fun1与fun2同时并行执行，依据结果选择性输出
设同步开销时间为R，则并行代码运行时间为R+max(t1+t2)
当R+max(t1+t2) < t1+t2*P时，可提升效率

if分支难以并行，此处将判断语句延后进行，先计算再判断(俗称莽夫执行，干了再说😏)

int c1, c2;
#pragma omp parallel sections
{
    #pragma omp section
    {c1=fun1(...);}
    #pragma omp section
    {c2=fun2(...);};
}

if (c1>c2)
    return c1+c2;
else return c2;
